## Архитектура

```
┌─────────────────────────────────────────────────────────┐
│                    Веб-интерфейс                        │
│               (index.html, порт 8081)                   │
└──────────────────────┬──────────────────────────────────┘
                       │ HTTP / SSE
┌──────────────────────▼──────────────────────────────────┐
│                  FastAPI (main.py)                       │
│   /api/upload  /api/batch/*/progress  /api/model/*      │
└──────┬─────────────┬──────────────────┬─────────────────┘
       │             │                  │
       ▼             ▼                  ▼
  batch_processor  pipeline.py    model_manager.py
  (очередь PDF)   (VLM-пайплайн)  (загрузка/выгрузка GPU)
       │             │                  │
       ▼             ▼                  ▼
  pdf_utils.py    vlm_client.py    start_vllm.sh
  (PDF → PNG)    (HTTP → vLLM)    (процесс vLLM)
                     │
                     ▼
            ┌────────────────┐
            │  vLLM (8001)   │
            │  Qwen3-VL-8B   │
            │  RTX 3090 GPU  │
            └────────────────┘
```

**Два отдельных процесса:**
- **FastAPI** (порт 8081) — веб-сервер, API, координация обработки
- **vLLM** (порт 8001) — inference-сервер модели, OpenAI-совместимый API

FastAPI управляет жизненным циклом vLLM через `model_manager.py`: запуск/остановка процесса, мониторинг статуса.

## Установка и запуск

### Требования

- Python 3.12+
- NVIDIA GPU с 16+ GB VRAM (рекомендуется RTX 3090 / A5000+)
- CUDA 12.x
- ~10 GB свободного места (модель + зависимости)

### Установка

```bash
cd /home/testusr4/QwenScan
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
```

Модель скачивается отдельно с HuggingFace:

```bash
pip install huggingface_hub
huggingface-cli download cyankiwi/Qwen3-VL-8B-Instruct-AWQ-8bit \
  --local-dir models/Qwen3-VL-8B-Instruct-AWQ-8bit
```

### Запуск

**1. Запуск vLLM (inference-сервер):**

```bash
bash start_vllm.sh
# Или через nohup:
nohup bash start_vllm.sh > vllm.log 2>&1 &
```

Загрузка модели занимает ~30–60 секунд. Готовность проверяется по `http://localhost:8001/v1/models`.

**2. Запуск FastAPI (веб-сервер):**

```bash
bash start_app.sh
# Или через nohup:
nohup bash start_app.sh > uvicorn.log 2>&1 &
```

**3. Доступ к интерфейсу:**

- Локально: `http://localhost:8081`
- Через SSH-туннель (если порт закрыт файрволом):
  ```bash
  ssh -L 8081:localhost:8081 testusr4@<server-ip>
  # Затем открыть http://localhost:8081
  ```

> vLLM можно не запускать заранее — есть кнопка загрузки модели в веб-интерфейсе.

## Конфигурация

Файл `app/config.py`:

| Параметр | Значение | Описание |
|---|---|---|
| `VLLM_BASE_URL` | `http://localhost:8001/v1` | Адрес vLLM API |
| `MODEL_NAME` | `document-parser` | Имя модели (served-model-name) |
| `MAX_CONCURRENT_REQUESTS` | `10` | Макс. параллельных запросов к VLM |
| `PDF_DPI` | `280` | DPI конвертации PDF → PNG |
| `EXTRACTION_MAX_TOKENS` | `1536` | Макс. токенов в ответе модели |
| `REQUEST_TIMEOUT` | `120.0` | Таймаут запроса к VLM (сек) |
| `TEMPERATURE` | `0.0` | Температура генерации (0 = детерминированно) |

Параметры vLLM (`start_vllm.sh`):

| Параметр | Значение | Описание |
|---|---|---|
| `--max-model-len` | `16384` | Макс. длина контекста (токены) |
| `--gpu-memory-utilization` | `0.7` | Доля VRAM (~17 GB из 24) |
| `--dtype` | `float16` | Тип данных весов |
| `--enforce-eager` | — | Без CUDA Graph (экономия VRAM) |

## API

### Загрузка файлов

```
POST /api/upload
Content-Type: multipart/form-data
```

| Поле | Тип | Описание |
|---|---|---|
| `files` | `File[]` | PDF-файлы |
| `custom_prompt` | `string` | Пользовательский промпт (опционально) |

Ответ:
```json
{"batch_id": "a1b2c3d4", "files": 5}
```

### Прогресс обработки (SSE)

```
GET /api/batch/{batch_id}/progress
```

Server-Sent Events с данными:
```json
{
  "status": "processing",
  "total_pages": 12,
  "pages_done": 7,
  "files": [
    {"filename": "doc.pdf", "status": "processing", "pages": 3, "pages_done": 2}
  ]
}
```

Статусы файла: `pending` → `converting` → `processing` → `done` / `error`
Статусы батча: `pending` → `processing` → `done` / `error`

### Результаты

```
GET /api/batch/{batch_id}/results
```

Ответ:
```json
{
  "batch_id": "a1b2c3d4",
  "status": "done",
  "results": [
    {
      "filename": "invoice_01.pdf",
      "тип_документа": "УПД",
      "номер_документа": "123",
      "дата_документа": "15.01.2026",
      "покупатель": "ООО Ромашка",
      "покупатель_инн": "7723022111",
      ...
    }
  ]
}
```

### Генерация Excel (по запросу)

```
POST /api/batch/{batch_id}/generate-excel
```

Ответ:
```json
{"ok": true, "filename": "batch_a1b2c3d4_20260213_120000.xlsx"}
```

### Скачивание Excel

```
GET /api/batch/{batch_id}/excel
```

Возвращает `.xlsx` файл.

### Просмотр загруженного PDF

```
GET /api/files/{batch_id}/{filename}
```

### Управление моделью

```
GET  /api/model/status   → {"status": "running|stopped|starting", "pid": 1234, "model": "document-parser"}
POST /api/model/load     → {"ok": true, "message": "Модель загружена", "pid": 1234}
POST /api/model/unload   → {"ok": true, "message": "Модель выгружена, GPU освобождена"}
```

## Веб-интерфейс

Одностраничное приложение (SPA) в `app/templates/index.html`. Тёмная тема, двухпанельная компоновка.

**Левая панель — загрузка и обработка:**
- Drag-and-drop зона для PDF
- Список загруженных файлов с превью
- Текстовое поле для пользовательского промпта
- Кнопка «Обработать» с прогресс-баром (SSE)

**Правая панель — результаты:**
- Таблица с извлечёнными данными (динамические колонки)
- Кнопка «Экспорт в Excel»

**Навбар:**
- Логотип и название
- Бейдж модели (VLM 8B-AWQ)
- Индикатор статуса модели (зелёный/серый кружок)
- Кнопка загрузки/выгрузки модели

## Режимы работы

### Шаблонный режим (по умолчанию)

Используется когда поле «Промпт» пустое. Извлекает фиксированный набор полей из бухгалтерских документов:

| Поле | Описание |
|---|---|
| `тип_документа` | УПД, счёт-фактура, товарная накладная, акт и т.д. |
| `номер_документа` | Номер (исправление О→0, удаление пробелов) |
| `дата_документа` | Дата в формате ДД.ММ.ГГГГ |
| `покупатель` | Наименование покупателя |
| `покупатель_инн` | ИНН покупателя (10 или 12 цифр) |
| `покупатель_кпп` | КПП покупателя (9 цифр, у ИП — пусто) |
| `покупатель_адрес` | Юридический адрес покупателя |
| `продавец` | Наименование продавца |
| `продавец_инн` | ИНН продавца |
| `продавец_кпп` | КПП продавца |
| `продавец_адрес` | Юридический адрес продавца |
| `ндс` | Сумма НДС (из строки ИТОГО) |
| `сумма_с_ндс` | Итоговая сумма с НДС |

**Пост-обработка шаблонного режима** (pipeline.py):
- Разделение объединённых ИНН/КПП (формат `7723022111/772201001`)
- Удаление КПП у ИП (индивидуальных предпринимателей)
- Валидация КПП (9 цифр, код региона 01–99)
- Очистка адресов от банковских реквизитов и телефонов
- Проверка ИНН по контрольным суммам (validators.py)
- Специальная обработка актов (нет продавца)
- Проверка НДС ≤ сумма с НДС

### Гибкий режим (с пользовательским промптом)

Пользователь вводит произвольные инструкции. Модель сама определяет набор полей на основе промпта.

Поддерживает нумерованные списки (`товары_наименование_1`, `товары_наименование_2`, ...).

## Обработка документов

### Пайплайн одной страницы

```
PDF-страница → PNG (280 DPI) → base64 → vLLM Chat API → JSON → пост-обработка → результат
```

1. **Конвертация**: PyMuPDF рендерит страницу в PNG с DPI=280
2. **Отправка в VLM**: PNG кодируется в base64, отправляется в OpenAI-совместимый API vLLM
3. **Парсинг**: Ответ модели (JSON в тексте) извлекается, удаляются теги `<think>`, markdown-обёртки, исправляются trailing commas
4. **Пост-обработка**: Коррекция ИНН/КПП, адресов, номеров документов
5. **Валидация**: Проверка контрольных сумм ИНН, формата КПП

### Пакетная обработка

Файлы обрабатываются группами по 2 PDF (настраивается через `PDF_BATCH_SIZE`):

```
[PDF_1, PDF_2] → конвертация в PNG → VLM → освобождение памяти → [PDF_3, PDF_4] → ...
```

Это позволяет обрабатывать тысячи файлов без переполнения RAM. Страницы внутри группы обрабатываются параллельно (до `MAX_CONCURRENT_REQUESTS` одновременных запросов к VLM).

### Объединение многостраничных документов

Если PDF содержит несколько страниц:
- Страницы без данных (только подписи/печати) пропускаются
- Первая страница с данными — базовая
- Последующие страницы дополняют недостающие поля
- **Суммы (НДС, сумма с НДС)** всегда берутся с последней страницы (там ИТОГО)
- Нумерованные товарные позиции объединяются (перенумерация)

## Управление моделью

Модель можно загружать/выгружать через веб-интерфейс или API.

**Состояния:**
- `running` — модель загружена, GPU занята (~17 GB VRAM)
- `stopped` — модель выгружена, GPU свободна (~400 MB)
- `starting` — идёт загрузка модели (30–60 сек)

**Процесс загрузки:**
1. `model_manager.py` запускает `start_vllm.sh` как subprocess
2. Ожидает готовности (polling `/v1/models` до 90 сек)
3. Возвращает статус в UI

**Процесс выгрузки:**
1. Находит PID процесса vLLM через `pgrep`
2. Отправляет SIGTERM, ждёт до 15 сек
3. При необходимости — SIGKILL
4. GPU полностью освобождается

## Экспорт в Excel

Генерация Excel вызывается по запросу (кнопка «Экспорт в Excel» после завершения обработки).

### Шаблонный режим

Фиксированная структура из 15 колонок:

| # | Колонка |
|---|---|
| 1 | Путь к файлу (гиперссылка на PDF) |
| 2 | Имя файла |
| 3 | Тип документа |
| 4 | Номер документа |
| 5 | Дата документа |
| 6 | Покупатель |
| 7 | ИНН покупателя |
| 8 | КПП покупателя |
| 9 | Адрес покупателя |
| 10 | Продавец |
| 11 | ИНН продавца |
| 12 | КПП продавца |
| 13 | Адрес продавца |
| 14 | НДС |
| 15 | Сумма с НДС |

Шапка — синяя (RGB: 4472C4), первая строка заморожена.

### Гибкий режим

Динамические колонки на основе извлечённых полей. Вложенные структуры разворачиваются с разделителем `→`. Массивы объединяются через `;`.

Ширина колонок подбирается автоматически (макс. 60 символов).
